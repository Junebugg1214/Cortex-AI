Why Cortex was made

The problem is not “AI memory.” The problem is memory continuity.

Teams and individuals are already teaching tools how they work. The failure is that this context is usually stored in product-specific silos, copied by hand, or trusted without review.

Before Cortex

  • Each assistant starts from scratch or from stale instructions
  • Important context is hidden in chats, exports, and project files
  • Switching tools means rebuilding the same memory over and over
  • There is no clear diff, blame, review, or rollback for memory changes

After Cortex

  • Your persistent Mind lives on disk, not inside one vendor product
  • Memory changes can be branched, reviewed, merged, and restored
  • Specialist Brainpacks attach only when the task actually needs them
  • Each runtime gets the right slice instead of one giant generic blob
Core model

Cortex is built around one top-level object: the Mind.

The Mind turns Cortex’s product surfaces into one understandable system: portable AI context, Brainpacks, and Git-style memory history all become properties of the same durable object.

How it works

From raw context to runtime-ready memory

Cortex does not just store notes. It composes, governs, and routes memory so the right context arrives in the right place at the right time.

01

Ingest

Scan tools, imports, repos, notes, and chats into durable memory candidates.

02

Version

Track memory changes with commits, branches, review, diff, and rollback.

03

Compose

Blend base Mind state with attached Brainpacks, policy rules, and task routing.

04

Mount

Write target-specific context into local tools and self-hosted runtimes.

$ cortex init
$ cortex mind remember self "We use TypeScript and Supabase."
$ cortex mind compose self --to codex --task "product strategy"
$ cortex mind mount self --to codex cursor claude-code --task "launch prep"
Use cases

Different workflows need different memory behavior.

Cortex is useful when you need continuity across tools, safer long-running memory, or specialized context that can travel with the work.

For people building with multiple AI coding tools

Keep your stack, conventions, and active project state portable as you move between coding agents. Mount the same Mind into Codex, Cursor, Claude Code, MCP, Hermes, or OpenClaw without rewriting your background every time.

For self-hosted memory infrastructure

Run Cortex locally or as an operator-managed service. Keep the store on disk you control, use scoped API keys, and expose only the surfaces you intend to expose.

For specialist knowledge that should stay durable

Turn notes, repositories, and source files into Brainpacks, then attach them to a Mind so Cortex can route expert context only when the task actually calls for it.

For teams that want shared patterns without a hosted memory cloud

Cortex gives small teams a self-hosted way to manage namespaces, branch risky memory changes, review them, and keep durable context aligned across workflows.
